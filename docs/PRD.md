# PRD: 음반 LP 전문 이커머스 백엔드

| 항목 | 값 |
|---|---|
| 프로젝트명 (가칭) | Groove — LP 전문 셀렉트샵 백엔드 |
| 버전 | 1.0 |
| 작성일 | 2026-05-05 |
| 진행 기간 | 12주 (단독) |
| 스택 | Java 21, Spring Boot 4.0.x, MySQL 8, Docker Compose |
| 산출물 | 백엔드 REST API + Postman 컬렉션 + 설계 문서 일체 |

---

## 1. 프로젝트 개요

음반 LP(Long Play, 바이닐) 전문 온라인 셀렉트샵의 백엔드 API. 일반 이커머스의 흐름(회원가입 → 상품 탐색 → 주문 → 결제 → 배송)을 갖추되, **LP 도메인 특성**(한정반, 단일 재고, 풍부한 음악 메타데이터)을 활용하여 동시성·검색 최적화 등 CS 기반 개선 사례를 자연스럽게 시연한다.

### 1.1 왜 LP 도메인인가
- **한정반/희귀반**이 흔한 시장이라 "소량 재고 + 동시 다발 주문" 시나리오가 도메인적으로 정당화됨 → 일반 이커머스의 "급조한 플래시 세일"보다 설득력 있는 동시성 시연
- **풍부한 메타데이터**(앨범·아티스트·장르·레이블·발매연도) → N+1 발생과 복합 인덱스 최적화 시연에 적합
- **단일 재고(1점) 희귀반** → 결제·재고 정합성 시연 임팩트 강화
- 일반 의류·잡화 이커머스와 차별화되어 포트폴리오 인상에 기여

## 2. 프로젝트 철학

본 프로젝트는 **"기술 자랑용 데모"가 아니라 "실제 운영 가능 수준의 완성된 백엔드"** 를 만드는 것을 목적으로 한다. 그 위에서 부하 테스트와 코드 측정으로 발견된 문제(N+1, 슬로우 쿼리, 동시성 충돌, 결제 정합성 등)를 CS 기반으로 해결하는 과정 자체를 차별화 요소로 삼는다.

### 핵심 원칙
1. **선(先) 동작, 후(後) 최적화.** 기능이 미완인 상태로 Redis/Kafka를 먼저 들이지 않는다.
2. **모든 도구는 "발견된 문제에 대한 응답"으로 도입한다.** 도구 사용 자체가 목적이 되지 않는다.
3. **수치 없는 개선은 개선이 아니다.** Before/After 비교를 항상 동반한다.
4. **트레이드오프를 명시한다.** 선택의 한계와 대안을 README에 기록한다.

## 3. 목표 및 비목표

### 3.1 목표
- 회원가입부터 배송 완료까지 LP 구매 흐름이 끊김 없이 동작하는 REST API
- 통합 테스트 기반의 신뢰성 있는 코드베이스 (라인 커버리지 60%+)
- 부하 테스트로 식별한 실제 병목을 해결하는 사례 1건 이상 (Before/After 그래프)
- 결제·주문 정합성 민감 영역에 대한 명확한 트랜잭션 경계 및 멱등성 처리

### 3.2 비목표
- 실제 PG사 연동 (모킹으로 처리)
- 분산 시스템 운영 (단일 인스턴스 + Docker Compose 가정)
- 프론트엔드 / UI
- 다음 기능군: 추천, Elasticsearch 검색, 쿠폰, 포인트, 위시리스트, 1:1 문의, 실시간 채팅, 소셜 로그인
- LP 도메인 특정 비목표:
  - 중고 LP 취급 (컨디션 등급별 가격 차별화 — v2 후보)
  - 예약 주문 (pre-order)
  - 트랙리스트 관리, 음원 미리듣기
  - 상품 옵션 (단일 SKU만 v1에서 다룸)
- "쓸 수 있는 모든 기술 다 쓰기"

| 역할 | 설명 | 인증 방식 |
|---|---|---|
| 일반 사용자 (USER) | LP를 탐색·구매하는 회원 | 이메일 + 비밀번호 |

| 관리자 (ADMIN) | 상품·주문 관리 | 이메일 + 비밀번호 (별도 권한) |

## 5. 핵심 사용자 흐름

### 5.1 회원 주문 흐름
1. 회원가입 (BCrypt 해싱, 이메일 인증 토큰은 발급만 — 메일 발송 모킹)
2. 로그인 → Access Token (TTL 30분) + Refresh Token (TTL 14일) 발급
3. LP 목록 / 상세 조회, 검색·필터링
4. 장바구니 담기 / 수량 변경 / 삭제
5. 주문 생성 (재고 검증 + 차감 → 주문 PENDING)
6. 결제 요청 (Idempotency-Key 헤더 포함) → 모킹 PG 호출
7. PG 웹훅 콜백 (자체 시뮬레이션) → 주문 PAID 전환 + 배송 엔트리 생성
8. 배송 상태 진행 (PREPARING → SHIPPED → DELIVERED, 스케줄러 시뮬레이션)
9. 배송 완료 후 리뷰/평점 작성 가능

### 5.2 게스트 주문 흐름
- 회원가입 없이 1~7 단계 진행 가능
- 주문 조회는 (주문번호 + 이메일) 매칭으로 인증

### 5.3 관리자 흐름
- 상품(LP) CRUD
- 아티스트 / 장르 / 레이블 CRUD
- 전체 주문 조회 및 상태 강제 전환
- 재고 조정

### 5.4 결제 실패 / 보상 트랜잭션 흐름
- PG 응답 실패: 주문 → PAYMENT_FAILED, 재고 복원
- 웹훅 미수신: 스케줄러 기반 재시도 (n분 후 PG 결제 상태 폴링)
- 멱등성 키 중복: 기존 결제 결과 그대로 반환

## 6. 기능 요구사항

### 6.1 인증/인가
- 이메일 + 비밀번호 회원가입 (BCrypt cost 12)
- Access / Refresh Token 분리 발급
- **Refresh Token Rotation**: 사용 시마다 새 토큰 발급, 이전 토큰 무효화. 재사용 감지 시 해당 사용자 전체 세션 무효화
- 토큰 저장은 v1에서 DB 기반, 추후 Redis 전환 가능한 인터페이스 유지
- 권한 분리: USER, ADMIN
- Spring Security `SecurityFilterChain` + JWT 필터

### 6.2 상품 (LP)
- 도메인 엔티티 분리: **Album / Artist / Genre / Label** (4개)
- LP(Album) 필드:
  - 앨범명, 아티스트(FK), 장르(FK), 레이블(FK)
  - 발매연도, 포맷 (12" LP / 더블 LP / 7" 싱글 등 enum)
  - 가격, 재고
  - 상태 (SELLING / SOLD_OUT / HIDDEN)
  - **한정반 여부** (boolean) — 시연 시나리오 핵심 플래그
  - 표지 이미지 URL
- 단일 SKU (컨디션·옵션 없음)

### 6.3 카테고리 / 분류
- **장르** 기반 단일 레벨 분류 (Rock, Jazz, Classical, K-Pop, Hip-Hop, Electronic, Indie 등)
- **아티스트** 별도 엔티티 — 아티스트 페이지 / 검색 가능
- **레이블** 별도 엔티티 — 검색 보조 (필수 노출은 아님)

### 6.4 검색
- 키워드 검색: 앨범명 + 아티스트명 동시 매칭
- 필터: 장르, 가격 범위, 발매연도 범위, 한정반 여부
- 정렬: 최신순 / 가격 오름차 / 가격 내림차 / 판매량
- 페이징: 오프셋 기반 (커서 기반은 발견 시 개선 후보)
- v1은 단순 LIKE + 인덱스 최적화로 처리, Elasticsearch 미도입

### 6.5 장바구니
- 회원 전용 (게스트는 즉시 구매 only)
- 상품 추가 / 수량 수정 / 삭제 / 비우기
- 장바구니 조회 (상품 정보 조인 결과 반환)

### 6.6 주문
- 다중 상품 주문 가능 (장바구니 일괄 또는 즉시 구매)
- 주문 생성 시 재고 검증 + 차감을 트랜잭션 내 처리
- 주문 상태 머신 (명시적 enum + 전이 검증):
  ```
  PENDING → PAID → PREPARING → SHIPPED → DELIVERED → COMPLETED
        ↘ CANCELLED
        ↘ PAYMENT_FAILED
  ```
- 주문 취소: PAID 이전, 또는 운영 정책상 PREPARING 단계까지 허용 (관리자만)

### 6.7 결제 (모킹)
- **자체 모킹 PG 모듈**: 별도 패키지 `payment.gateway.mock` 으로 분리
- **Strategy 패턴** 으로 PG 인터페이스 추상화 (`PaymentGateway` 인터페이스 → `MockPaymentGateway` 구현체) → 추후 실 PG 연동을 위한 확장 포인트
- 동작:
  - 결제 요청 → 처리 지연 시뮬레이션 (랜덤 100~500ms) → 성공/실패 응답 (성공률 설정 가능, 기본 95%)
  - 비동기 웹훅 콜백 시뮬레이션 (스케줄러로 1~5초 후 콜백 발사)
- **Idempotency-Key** 처리: 동일 키 재요청 시 기존 결과 반환 (DB에 키-결과 매핑 저장)
- 결제 실패 시 보상 트랜잭션: 주문 PAYMENT_FAILED + 재고 복원

### 6.8 배송
- 운송장 번호 발급 (UUID 기반, 외부 API 미연동)
- 상태: PREPARING → SHIPPED → DELIVERED
- 스케줄러 또는 Spring Application Event로 일정 시간 후 자동 상태 진행 (시연용)
- 배송지 정보 저장 (수령인, 주소, 연락처)
- LP 특성 반영: "안전 포장 요청" 단순 boolean 필드 (실제 운영 시뮬레이션)

### 6.9 관리자
- 상품(Album) CRUD
- 아티스트 / 장르 / 레이블 CRUD
- 주문 전체 조회 / 상태 강제 전환 / 환불 처리
- 재고 조정

### 6.10 리뷰 / 평점
- 배송 완료(DELIVERED) 주문에 한해 작성 가능
- 1~5점 + 텍스트
- 상품(Album)별 평균 평점 노출
- 1주문-1상품-1리뷰 제약 (DB 유니크 제약)

## 7. 비기능 요구사항

### 7.1 보안
- 비밀번호 BCrypt (cost 12)
- JWT Secret은 환경변수 분리, 코드/저장소 미포함
- Rate Limiting: 로그인·회원가입 등 민감 엔드포인트에 IP 기반 제한 (Bucket4j 또는 인터셉터 자체 구현)
- 입력 검증: `@Valid` + `@Validated` 전수 적용
- SQL Injection: JPA 사용으로 기본 보호, 동적 쿼리 시 파라미터 바인딩 강제
- 민감 정보 로깅 금지 (비밀번호, 토큰, 결제 카드정보)

### 7.2 데이터 정합성
- 모든 쓰기 작업은 명시적 트랜잭션 경계
- 결제·주문은 멱등성 키 기반 중복 방지
- DB 외래 키 제약 + 비즈니스 룰(상태 전이) 이중 검증

### 7.3 관측성
- Logback + MDC: 요청별 추적 ID(`X-Request-Id`) 부여, 모든 로그에 자동 포함
- 주요 비즈니스 이벤트 INFO 레벨 로깅 (주문 생성, 결제, 상태 변경)
- Spring Boot Actuator 활성화 (`/actuator/health`, `/actuator/metrics`)

### 7.4 테스트
- JUnit 5 + Spring Boot Test
- 통합 테스트: Testcontainers(MySQL) 우선, 도입 부담 시 H2
- 단위 테스트: 도메인 로직(상태 전이, 검증 규칙) 집중
- 커버리지 목표: 전체 라인 60%+, 핵심 도메인 80%+

### 7.5 성능 (측정 기반)
- **사전 목표 수치는 못 박지 않는다** — Before/After 상대 비교가 핵심
- 부하 테스트 도구: k6
- 시나리오: (1) 상품 검색 (2) 주문 생성 (3) 한정반 동시 주문 (4) 결제 처리

## 8. 차별화 시연 시나리오 (CS 개선 사례)

각 항목은 **"문제 발견 → 분석 → 개선 → 측정"** 사이클로 README에 기록한다. 모든 시연은 측정 데이터 동반 필수.

### 8.1 한정반 동시 주문 — 동시성 처리
- **시나리오**: "BTS 데뷔 10주년 한정반 LP 100장" 상품에 동시 1000 요청
- **단계**:
  1. 단순 구현 (`stock -= qty`) → 오버셀 재현, 로그/스크린샷 보존
  2. 비관적 락 (`SELECT ... FOR UPDATE`) → 정합성 OK, TPS 측정
  3. (필요 시) 낙관적 락 또는 Redis 분산락 비교 검토
- **산출물**: 단계별 TPS, 응답시간, 에러율 비교표

### 8.2 N+1 문제
- 상품 목록 조회 시 Album → Artist → Genre → Label 연관 조회에서 자연 발생
- `Hibernate Statistics`로 쿼리 카운트 측정
- 페치 조인 / `@EntityGraph` 적용 후 쿼리 수 비교 (예: 1+N → 1)

### 8.3 검색 슬로우 쿼리 → 인덱스
- 시드 데이터: 5~10만 건 (음악 메타데이터 활용 또는 자체 생성)
- 시나리오 쿼리: "Rock + 1970~1980년 + 5만원 이하 + 최신순"
- `EXPLAIN`으로 풀 스캔 확인 → 복합 인덱스 추가 → 응답 시간 / 스캔 row 수 비교

### 8.4 결제 멱등성 검증
- 동일 Idempotency-Key로 동시 2회 요청 시뮬레이션 → 단일 결제만 처리되는지 통합 테스트
- 웹훅 중복 수신 시 상태 전이 1회만 발생하는지 검증

### 8.5 (시간 여유 시) 단일 재고 희귀반
- 재고 1짜리 상품에 동시 100 요청 → 정확히 1건만 성공해야 함
- 가장 임팩트 큰 정합성 시연 — 면접에서 끌고 갈 무기
